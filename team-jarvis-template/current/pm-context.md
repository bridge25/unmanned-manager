## LLM 오케스트레이션 규칙

### Codex/Gemini → Worker 협업 패턴

**문제**: codex/gemini는 별도 프로세스로 실행되어 Worker Claude가 맥락을 모름

**해결**: PM이 중간에서 맥락 전달

```
[순차 협업 플로우]

1. PM → codex 실행 (tmux new-window + script)
2. PM ← codex 결과 수신
3. PM → Worker에게 맥락 포함 지시:

   "이전 단계에서 codex 린트 체크 결과:
   - 파일 A: 미사용 import 3개
   - 파일 B: 타입 오류 2개

   위 문제들을 수정해줘."
```

### 맥락 전달 규칙

| 상황 | PM이 해야 할 것 |
|------|----------------|
| codex → Worker | codex 결과 요약 + 수정 요청 |
| gemini 리서치 → Worker | 핵심 발견사항 + 적용 지시 |
| Worker A → Worker B | A의 작업 결과 요약 + B의 역할 명시 |

### 예시: 린트 → 수정 → 테스트

```
Step 1: PM → codex
  "린트 체크해줘"

Step 2: PM → Worker (codex 결과 포함)
  "codex 린트 결과:
   - src/main.py: line 15 unused import
   - src/utils.py: line 42 type error
   이 문제들 수정해줘"

Step 3: PM → codex (수정 검증)
  "다시 린트 체크해줘"
```

### TTY 요구 도구 실행 방법

```bash
# codex/gemini는 TTY 필요 → tmux new-window + script
tmux new-window -t {session} -n {task_id} \
  "script -q {result_file} codex --full-auto '{instruction}'"

# 결과 대기
grep -q "DONE" {result_file}

# 결과 파싱 (ANSI 제거)
cat {result_file} | sed 's/\x1B\[[0-9;]*[a-zA-Z]//g'
```

## 🧠 MindCollab 댓글 기반 트래킹

> **Single Source of Truth + 댓글 대화 = 입체적 프로젝트 관리**

### 핵심 원칙
- **노드 상태**: MindCollab이 관리 (backlog → in_progress → done)
- **진행 내용**: 댓글로 Tony ↔ JARVIS 대화 기록
- **브리핑 시**: 노드 상태 + 최근 댓글 → 완전한 현황 파악

### 자비스 댓글 작성 규칙
```bash
# 작업 시작 시
mc comment N78 "작업 시작합니다. 예상 소요: 2-3시간" --author "JARVIS"

# 진행 중 체크포인트
mc comment N78 "Step 1 (테스트 실행) 완료. Step 2 진행 중" --author "JARVIS"

# 블로커 발견 시
mc comment N78 "⚠️ 블로커: GLM API 크레딧 부족. Tony님 확인 필요" --author "JARVIS"

# 작업 완료 시
mc comment N78 "완료. 결과물: 테스트 5개 통과, 2개 실패 (원인 분석 완료)" --author "JARVIS"
```

### 브리핑 시 댓글 활용
```bash
# 우선순위 높은 노드 댓글 확인
for node in N77 N78 N74 N88; do
  mc comments $node | grep -E "@Tony|@JARVIS"
done

# 댓글에서 추출할 정보:
# - Tony님의 최근 피드백 → 우선순위 조정
# - 블로커 → 즉시 해결 방안 제시
# - 완료된 체크포인트 → 다음 단계 진행
# - 질문/요청 → 브리핑에 포함
```
